title: "My First Merge Conflict"
title_ja: "初めてのマージ衝突"
domain: it
level: [3, 4]
format: dialogue
speakers: "Emma, Ken"
---

Emma: Ken, can you help me for a minute? Something looks wrong.

Ken: Sure, what's going on with your code?

Emma: I tried to merge my branch, but I got an error.

Ken: Let me see the screen. Oh, it's a merge conflict.

Emma: A merge conflict? I've never seen that before.

Ken: It just means two people changed the same file.

Emma: So both of us edited the same part?

Ken: Yes, and Git can't decide which change to keep.

Emma: That makes sense. So what should I do now?

Ken: You open the file and look for the marks.

Emma: What kind of marks am I looking for?

Ken: Little arrows and lines. They show both versions.

Emma: Oh, I see them here. There's my code and yours.

Ken: Right. Now you pick the right one, or mix them.

Emma: I think we actually need both lines here.

Ken: Good eye. Then keep both, but remove the marks.

Emma: Should I delete these arrows too?

Ken: Yes, delete all of them. They aren't real code.

Emma: Okay, done. The file looks clean now.

Ken: Nice. Now save it and add the file again.

Emma: Like this? I'll run the add command first.

Ken: Perfect. Then commit, and the conflict is gone.

Emma: Wow, that was easier than I thought.

Ken: It scares people, but it's really not so bad.

Emma: I was worried I broke the whole project.

Ken: No, Git keeps everything safe. You can't lose much.

Emma: That's a relief. I'll test it before I push.

Ken: Smart. Always run the tests after a merge.

Emma: The tests all passed. Green across the board.

Ken: Great. Now push it and open your pull request.

Emma: Done. Thanks so much for walking me through this.

Ken: No problem. You'll handle the next one alone.

Emma: I hope so. At least now I know the steps.

Ken: Just pull often, and you'll get fewer conflicts.

Emma: Good tip. I'll pull first thing every morning.

Ken: That's the habit. Come find me if you're stuck.
